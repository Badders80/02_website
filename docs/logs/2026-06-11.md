## Decisions
- **WIF over service account keys**: GCP org policy blocks `iam.disableServiceAccountKeyCreation`, so WIF is the only viable path for Vercel→GCP auth
- **Cloud Run proxy as IAM bypass**: Cloud Run sits in front of Cloud Functions and forwards identity tokens + Firebase auth tokens; needed because GCP org policy blocks direct IAM on Cloud Functions
- **Two-header auth pattern**: Cloud Functions accept Firebase ID tokens from both `Authorization: Bearer` and `X-Firebase-Token` headers to support both direct calls and proxy forwarding
- **Bypass mode for local dev**: `NEXT_PUBLIC_BYPASS_AUTH_KYC` env var allows local testing without Firebase auth

## Narrative
This session tackled the GCP org policy blocker head-on. Root cause was that GCP's `iam.allowedPolicyMemberDomains` constraint prevents setting `allUsers` or `allAuthenticatedUsers` on any IAM policy — which is exactly what Cloud Functions and Cloud Run need for public API access. Since the org policy also blocks service account key creation, the only path was Workload Identity Federation. Built the full WIF infrastructure: pool, OIDC provider (linked to Vercel's issuer), and service account with impersonation grants. Also created a Cloud Run proxy that sits in front of Cloud Functions, holding the identity token and forwarding Firebase auth tokens from the browser. On the frontend, updated the API client to route through the proxy and added "Apply to Own" button to marketplace listings. Everything is deployed and ready — the last piece is enabling Vercel OIDC in the Vercel dashboard, which makes the `VERCEL_OIDC_TOKEN` env var available to serverless functions.
